## 🚀 Descripción General

El **Sistema Integrado Blatam Academy** es una plataforma completa que integra múltiples servicios de IA y automatización empresarial en una arquitectura unificada y escalable.

### ✨ Características Principales

- 🤖 **50+ Modelos de IA** integrados y listos para usar
- 🏗️ **Arquitectura Modular** con microservicios independientes
- ⚡ **Ultra Adaptive KV Cache Engine** para máximo rendimiento
- 🔄 **Integración Completa** con servicios externos (Spotify, OpenAI, NotebookLM, etc.)
- 📊 **Monitoreo en Tiempo Real** con Prometheus y Grafana
- 🔒 **Seguridad Empresarial** con autenticación JWT y rate limiting
- 📈 **Escalabilidad Horizontal** con Docker y Kubernetes
- 🧪 **Testing Completo** con cobertura E2E

### 📊 Patrones Arquitectónicos

| Patrón | Descripción | Implementación | Beneficios |
|--------|-------------|----------------|------------|
| **Microservicios** | Servicios independientes y desacoplados | Cada feature es un microservicio | Escalabilidad independiente, despliegue independiente |
| **API Gateway** | Punto único de entrada | Integration System (Puerto 8000) | Enrutamiento centralizado, autenticación unificada |
| **Service Discovery** | Descubrimiento automático de servicios | Consul/etcd, Docker DNS | Configuración dinámica, alta disponibilidad |
| **Circuit Breaker** | Prevención de fallos en cascada | Resilience4j, Polly | Resiliencia, degradación elegante |
| **Event Sourcing** | Almacenamiento de eventos | Redis Streams, Kafka | Auditoría completa, replay de eventos |
| **CQRS** | Separación de lectura/escritura | PostgreSQL (Write) + Redis (Read) | Optimización de queries, escalabilidad |
| **Saga Pattern** | Transacciones distribuidas | Choreography, Orquestación | Consistencia eventual, transacciones complejas |
| **Bulkhead** | Aislamiento de recursos | Docker containers, Thread pools | Aislamiento de fallos, control de recursos |
| **Retry Pattern** | Reintentos automáticos | Exponential backoff | Resiliencia ante fallos temporales |
| **Health Check** | Monitoreo de salud | `/health` endpoints | Detección temprana de problemas |
| **Rate Limiting** | Control de tasa de requests | Token bucket, Sliding window | Protección contra abuso, estabilidad |
| **Load Balancing** | Distribución de carga | Nginx, Kubernetes | Alta disponibilidad, escalabilidad |

### 🔄 Comunicación entre Servicios

#### Sincrónica (REST/HTTP)

```python
# Ejemplo de comunicación REST entre servicios
import httpx

async def call_service(service_url: str, endpoint: str, data: dict):
    async with httpx.AsyncClient(timeout=30.0) as client:
        response = await client.post(
            f"{service_url}/{endpoint}",
            json=data,
            headers={"Authorization": f"Bearer {token}"}
        )
        return response.json()
```

#### Asíncrona (Event-Driven)

```python
# Ejemplo de comunicación asíncrona con Redis Streams
import redis.asyncio as redis

async def publish_event(stream: str, event: dict):
    r = await redis.from_url("redis://localhost:6379")
    await r.xadd(stream, event)

async def consume_events(stream: str, consumer_group: str):
    r = await redis.from_url("redis://localhost:6379")
    while True:
        messages = await r.xreadgroup(
            consumer_group, "consumer-1", {stream: ">"}, count=10
        )
        for stream, messages_list in messages:
            for msg_id, data in messages_list:
                await process_event(data)
```

#### Ejemplo de Circuit Breaker

```python
from tenacity import retry, stop_after_attempt, wait_exponential
from circuitbreaker import circuit

@circuit(failure_threshold=5, recovery_timeout=60)
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=4, max=10)
)
async def call_external_service(url: str, data: dict):
    async with httpx.AsyncClient(timeout=10.0) as client:
        response = await client.post(url, json=data)
        response.raise_for_status()
        return response.json()
```

### 🎯 Decisiones de Diseño Arquitectónico

#### ¿Por qué Microservicios?

- **Escalabilidad Independiente**: Cada servicio puede escalarse según su carga
- **Tecnología Heterogénea**: Diferentes servicios pueden usar diferentes stacks
- **Despliegue Independiente**: Actualizaciones sin afectar otros servicios
- **Falla Aislada**: Un servicio caído no afecta todo el sistema
- **Equipos Autónomos**: Equipos pueden trabajar independientemente

#### ¿Por qué API Gateway?

- **Punto Único de Entrada**: Simplifica el cliente
- **Cross-Cutting Concerns**: Autenticación, logging, rate limiting centralizados
- **Protocol Translation**: HTTP, gRPC, WebSocket en un solo punto
- **Service Aggregation**: Combinar respuestas de múltiples servicios

#### ¿Por qué Event-Driven?

- **Desacoplamiento**: Servicios no necesitan conocerse directamente
- **Escalabilidad**: Procesamiento asíncrono paralelo
- **Resiliencia**: Mensajes persisten si un servicio está caído
- **Auditoría**: Historial completo de eventos

### 🔐 Seguridad y Autenticación

```
┌─────────────┐
│   Cliente   │
└──────┬──────┘
       │ 1. Request
       ▼
┌─────────────────┐
│  Nginx (SSL)    │ ──► Certificados TLS/SSL
│                 │ ──► DDoS Protection
│                 │ ──► IP Whitelisting
└──────┬──────────┘
       │ 2. Forward
       ▼
┌─────────────────┐
│ Integration     │
│ System          │ ──► JWT Authentication
│ (API Gateway)   │ ──► Rate Limiting
│                 │ ──► CORS
│                 │ ──► Request Validation
└──────┬──────────┘
       │ 3. Validate
       ▼
┌─────────────────┐
│  Auth Service   │ ──► Token Validation
│                 │ ──► User Permissions
│                 │ ──► Role-Based Access
└──────┬──────────┘
       │ 4. Authorize
       ▼
┌─────────────────┐
│  Microservice   │ ──► Business Logic
│                 │ ──► Service-to-Service Auth
└─────────────────┘
```

#### Flujo de Autenticación Detallado

1. **Cliente → Nginx**: Request HTTPS con credenciales
2. **Nginx → Integration System**: Forward con headers originales
3. **Integration System**: 
   - Extrae token JWT del header
   - Valida token con Auth Service
   - Verifica permisos y roles
   - Aplica rate limiting
4. **Auth Service**: 
   - Valida firma del token
   - Verifica expiración
   - Consulta permisos en DB
5. **Integration System → Microservice**: 
   - Agrega headers de autenticación internos
   - Enruta request al microservicio
6. **Microservice**: 
   - Valida autenticación interna
   - Ejecuta lógica de negocio
   - Retorna respuesta

### ➕ Agregar Nuevo Servicio

Sigue estos pasos para agregar un nuevo servicio al sistema:

#### 1. Crear Estructura del Servicio

```bash
# Crear directorio del servicio
mkdir nuevo_servicio
cd nuevo_servicio

# Estructura recomendada
mkdir -p api core services models utils config tests
touch main.py requirements.txt README.md
```

#### 2. Implementar API FastAPI

```python
# nuevo_servicio/main.py
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI(title="Nuevo Servicio", version="1.0.0")

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

@app.get("/health")
async def health():
    return {"status": "healthy"}

@app.post("/api/v1/process")
async def process(data: dict):
    # Lógica del servicio
    return {"result": "processed"}

if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8010)
```

#### 3. Agregar al Docker Compose

```yaml
# docker-compose-all.yml
nuevo_servicio:
  build: ./nuevo_servicio
  container_name: nuevo_servicio
  ports:
    - "8010:8000"
  environment:
    - ENVIRONMENT=production
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
    interval: 30s
    timeout: 10s
    retries: 3
  restart: unless-stopped
```

#### 4. Configurar en Integration System

```python
# integration_system/config/services.py
SERVICES = {
    "nuevo_servicio": {
        "url": "http://nuevo_servicio:8000",
        "timeout": 30,
        "retries": 3
    }
}
```

#### 5. Actualizar Nginx


### 🔐 Autenticación y Autorización

#### JWT Tokens

```python
# Generar token
from jose import jwt
from datetime import datetime, timedelta

def create_access_token(data: dict, expires_delta: timedelta):
    to_encode = data.copy()
    expire = datetime.utcnow() + expires_delta
    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt

# Validar token
def verify_token(token: str):
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        return payload
    except JWTError:
        return None
```

### 🛡️ Headers de Seguridad

```python
# middleware/security.py
from fastapi.middleware.trustedhost import TrustedHostMiddleware

app.add_middleware(
    TrustedHostMiddleware,
    allowed_hosts=["*.blatam.academy", "localhost"]
)

# Headers de seguridad
@app.middleware("http")
async def add_security_headers(request: Request, call_next):
    response = await call_next(request)
    response.headers["X-Frame-Options"] = "DENY"
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-XSS-Protection"] = "1; mode=block"
    response.headers["Strict-Transport-Security"] = "max-age=31536000"
    response.headers["Content-Security-Policy"] = "default-src 'self'"
    return response
```

### ⚠️ Problemas Comunes y Soluciones

#### 1. Servicio no inicia

**Síntomas:**
- El servicio no responde
- Error en logs de Docker
- Health check falla

**Soluciones:**

```bash
# Ver logs detallados
docker-compose logs [service-name] --tail=100

# Verificar estado del contenedor
docker-compose ps [service-name]

# Reiniciar servicio
docker-compose restart [service-name]

# Verificar recursos disponibles
docker stats

# Verificar puertos en uso
netstat -tulpn | grep :8000

# Reconstruir servicio
docker-compose build [service-name]
docker-compose up -d [service-name]
```

**Causas comunes:**
- Puerto ya en uso
- Memoria insuficiente
- Variables de entorno incorrectas
- Dependencias no disponibles

#### 2. Error de conexión a Base de Datos

**Síntomas:**
- Timeout al conectar a PostgreSQL
- Error "connection refused"
- Error de autenticación

**Soluciones:**

```bash
# Verificar que PostgreSQL está corriendo
docker-compose ps postgres

# Reiniciar PostgreSQL
docker-compose restart postgres

# Verificar conexión
docker-compose exec postgres psql -U postgres -c "SELECT 1;"

# Verificar variables de entorno
docker-compose exec postgres env | grep DATABASE

# Verificar logs de PostgreSQL
docker-compose logs postgres --tail=50

# Verificar configuración de conexión
docker-compose exec integration_system env | grep DATABASE_URL

# Probar conexión manual
psql -h localhost -U postgres -d blatam_academy
```

**Causas comunes:**
- PostgreSQL no iniciado
- Credenciales incorrectas
- Puerto incorrecto
- Red Docker no configurada

#### 3. Memoria Insuficiente

**Síntomas:**
- Servicios se cierran inesperadamente
- OOM (Out of Memory) errors
- Sistema lento

**Soluciones:**

```bash
# Verificar uso de memoria
docker stats
free -h
df -h

# Limpiar Docker
docker system prune -a
docker volume prune

# Reiniciar servicios
docker-compose down && docker-compose up -d

# Limpiar caché del KV Cache
python bulk/core/ultra_adaptive_kv_cache_cli.py clear-cache

# Reducir límites de memoria en docker-compose
# Agregar a cada servicio:
deploy:
  resources:
    limits:
      memory: 2G
    reservations:
      memory: 1G
```

**Prevención:**
- Monitorear uso de memoria
- Configurar límites apropiados
- Limpiar cache regularmente

#### 4. KV Cache - Alto Uso de Memoria

**Síntomas:**
- Memoria RAM agotada
- Sistema lento
- OOM errors

**Soluciones:**

```python
from bulk.core.ultra_adaptive_kv_cache_config_manager import ConfigManager, ConfigPreset

# Reducir tamaño de caché
config_manager = ConfigManager(engine)
await config_manager.update_config('cache_size', 4096)  # Reducir a 4GB

# Usar preset memory_efficient
ConfigPreset.apply_preset(engine, 'memory_efficient')

# Cambiar política de evicción
await config_manager.update_config('eviction_policy', 'lru')

# Limpiar cache manualmente
await engine.clear_cache()
```

## 🤝 Contribución

¡Las contribuciones son bienvenidas! Este proyecto es mantenido por la comunidad y agradecemos cualquier contribución.

> 📖 **Guía Completa**: Consulta nuestra [Guía de Contribución](CONTRIBUTING.md) para detalles completos.

### 🎯 Cómo Contribuir

#### 1. Preparar el Entorno

```bash
# Fork el repositorio en GitHub
# Clonar tu fork
git clone https://github.com/tu-usuario/blatam-academy.git
cd blatam-academy/agents/backend/onyx/server/features

# Crear branch para tu feature
git checkout -b feature/mi-nueva-feature

# Instalar dependencias de desarrollo
pip install -r requirements-dev.txt
pre-commit install
```

#### 2. Hacer Cambios

```bash
# Hacer tus cambios
# Asegúrate de seguir las convenciones:
# - Código formateado con Black
# - Tests pasando
# - Documentación actualizada

# Verificar código
black .
flake8 .
mypy .
pytest
```

#### 3. Commit y Push

```bash
# Commit con mensaje descriptivo
git commit -m "feat: agregar nueva funcionalidad X"

# Push al branch
git push origin feature/mi-nueva-feature
```

#### 4. Crear Pull Request

1. Ve a GitHub y crea un Pull Request
2. Completa la plantilla de PR
3. Espera revisión del equipo
4. Responde a comentarios y sugerencias

### 📋 Tipos de Contribuciones

| Tipo | Descripción | Ejemplos |
|------|-------------|----------|
| **🐛 Bug Fix** | Corrección de errores | `fix: corregir error en autenticación` |
| **✨ Feature** | Nueva funcionalidad | `feat: agregar endpoint de análisis` |
| **📚 Docs** | Mejoras de documentación | `docs: actualizar guía de instalación` |
| **⚡ Performance** | Optimizaciones | `perf: optimizar queries de base de datos` |
| **♻️ Refactor** | Refactorización de código | `refactor: reorganizar estructura de módulos` |
| **🧪 Tests** | Agregar o mejorar tests | `test: agregar tests para nuevo servicio` |
| **🎨 Style** | Formato y estilo | `style: formatear código con Black` |

### 🔍 Checklist de Contribución

Antes de crear un PR, asegúrate de:

- [ ] Código sigue las convenciones del proyecto
- [ ] Tests pasan (`pytest`)
- [ ] Código formateado (`black .`)
- [ ] Sin errores de linting (`flake8 .`)
- [ ] Documentación actualizada
- [ ] Commits con mensajes descriptivos
- [ ] PR tiene descripción clara
- [ ] No hay conflictos con main

### 🏷️ Convenciones de Commits

Usa [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Ejemplos:**
```bash
feat(api): agregar endpoint de análisis de documentos
fix(cache): corregir memory leak en KV Cache
docs(readme): actualizar sección de instalación
perf(db): optimizar query de usuarios
```
